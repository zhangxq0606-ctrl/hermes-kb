# Xq.KB 项目规范

## 知识库结构

```
d:\hermes-kb\kb\
├── 000_Dashboard.md          # 首页仪表板
├── core/
│   ├── insight/              # 深度认知（精炼后的洞察）
│   ├── note/                 # 零散灵感
│   └── question/             # 核心问题
├── manual/
│   └── technical/            # 技术知识库
├── raw/                      # 原始材料（采集的原文）
├── inbox/                    # 新文件入口
├── processing/               # 事务处理中
├── scripts/                  # 管线脚本
└── api/                      # Web 服务层
```

## 管线机制

- `hermes_engine.py` — AI 分类/提纯/分裂
- `index_guard.py` — 增量索引
- `weekly_scan.py` — 周度审计和问题生成
- Flat Web 服务是手机端访问入口，不干扰它

## 检索策略

**Claude Code 是首选检索入口。** 当用户提问需要查知识库时，按以下优先级读取：

1. **先查 `core/topic/`** — 编译后的主题页（综合结论），这是最快获取全貌的入口
2. **再查 `core/insight/_refined.md` / `manual/technical/_refined.md`** — 单篇精炼笔记
3. **最后查 `raw/`** — 原始材料，仅在需要核对事实来源时查阅

问答引擎（/chat、/ask）是移动端应急通道。

## 协作规范

- **启动任务先查skill**：每项任务开始时，先检查 `.claude/skills/` 下有无匹配的 skill 文件，有则按 skill 定义的工作流执行，而非凭默认行为操作。
- **主题探讨类任务**：用户输出大概主题后，先与用户思路探讨、延伸梳理，达成共识后再落笔成文，而非直接写出一版成品。探讨时聚焦核心思路，不要纠结概念定义或钻牛角尖式追问。
- **输出分支判定**：用户表明写作意图（如"写公众号"）时，讨论充分后自动在合适节点分两条线询问——① **入库**：要不要存为知识库精炼版？② **写作**：要不要输出最终正文？两者可先后、可同时，由用户决定。输出后进入**用户润色 → 用户敲定发布版 → 入库最终版**的流程。

## SAFEGUARD

- 不允许删除已有文件
- 不允许修改 raw/ 下的原始文件
- 允许在 core/insight/ 和 manual/technical/ 下新建 `_refined.md`，但重写已有 refined 需用户明确确认

## 文件写入规范（CC 直写服务器时适用）

你在服务器端写入知识文件时，必须遵循以下格式，否则静态站无法正确解析展示。

### 精炼笔记 → core/insight/ 或 manual/technical/

必须成对创建两个文件：

#### {主题}_refined.md（精炼版）

格式：

`markdown
---
related:
- file: 相关笔记文件名
  title: 相关笔记标题
---

🎯 【一句话洞察】
- 一句话说透本质

🗺️ 【核心逻辑链】
1. 逻辑点1
2. 逻辑点2

📦 【金句与底层案例夹囊】
>  引用的金句

[相关：xxx](xxx.md)

> ⏱️ 时效性: high

> 标签: principle
`

各项说明：
- 最前面 YAML frontmatter 用 \---\ 包裹，related 可选
- \🎯\ 后面必须是 \- 一句话\ 列表项格式，静态站取它作卡片摘要
- \🗺️\ 和核心逻辑链必填
- \📦\ 金句可选
- \> 标签:\ 必须顶格，标签值严格用小写英文

标签可选值：

| 标签 | 含义 |
|---------|------|
| principle | 认知/方法论 |
| tool | 工具型资源 |
| case | 案例/复盘 |
| reflection | 自我反思 |
| reference | 参考资料 |
| opinion | 观点/评论 |

#### {主题}.md（原文）

- 普通 markdown，保留原始表述
- 无格式要求

#### 命名规范
- 中文命名，下划线连接：产品思维的打磨_refined.md
- 不要用拼音

### 写作成品 → writing/

- 无结构要求，自由格式
- 不需要成对，单文件即可

### 禁止写入的目录
- 不要在服务器写 inbox/ 或 processing/，走不通管线
- 不要写 raw/，那是原文冷备区
