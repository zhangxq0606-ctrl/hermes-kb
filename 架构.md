# Hermes 架构

## 技术栈

| 层 | 选型 | 说明 |
|----|------|------|
| 语言 | Python 3.13 | 本地运行，无外部依赖框架 |
| AI | 阿里百炼 DashScope (qwen3.6-flash) | 分类、提纯、问题生成、问答 |
| 同步 | Syncthing v2.2.0 + GitHub（中转站）| 手机↔电脑原子文件同步 + 两端git同步 |
| 索引 | JSON 文件 + SHA256 | 轻量增量索引，无数据库 |

> ⚠️ 动态 Web 服务（Flask + Cloudflare Tunnel → zs.captainxq.me）已于 2026-06-15 停用。前端浏览全部走 read.captainxq.me（Cloudflare Pages 静态站），交互走手机 CC 直连服务器。

## 模块划分

```
┌─────────────────────────────────────┐
│            main.py（管线调度）        │
├──────┬──────┬──────────┬────────────┤
│Bootstrap│Engine│Index Guard│Weekly Scan│
│ 启动自检 │ 分流 │ 增量索引  │  周度审计  │
└──────┴──────┴──────────┴────────────┘
```

| 模块 | 文件 | 职责 | 阻断性 |
|------|------|------|--------|
| 调度器 | `main.py` | 管线编排，JSON 最终报告 | — |
| 自检 | `scripts/bootstrap.py` | 目录/文件存在性、Syncthing 守护 | 阻断 |
| 引擎 | `scripts/hermes_engine.py` | 三阶段事务分流 + AI 分类/提纯/分裂/打标签 | 非阻断 |
| 编译器 | `scripts/compiler.py` | 周度编译：按语义+标签聚合同主题精炼，AI 合成主题页 | 非阻断 |
| 索引 | `scripts/index_guard.py` | SHA256 + mtime 增量索引 | 非阻断 |
| 周扫 | `scripts/weekly_scan.py` | 摘要审计 + Dashboard 问题队列 + 读已回答问题做上下文 | 非阻断 |

## 三端架构与同步机制

```
┌─────────────────────────────────────────────────────────────────────┐
│                         GitHub（中转站）                              │
│  两端不直连，所有交换通过 git push / git pull 完成，无 scp、无 ssh   │
└───────┬─────────────────────────────────────┬───────────────────────┘
        │                                     │
        ▼                                     ▼
┌───────────────────┐             ┌──────────────────────────┐
│  本地 Windows       │             │  云服务器（阿里云 2C2G）  │
│                     │             │                          │
│  watch_inbox.ps1    │             │  sync.sh（cron */1）     │
│  ├─ 监听 inbox/     │             │  ├─ git stash+rebase+push│
│  ├─ 60s 防抖       │             │  ├─ build_static         │
│  ├─ 触发引擎+建站   │             │  └─ read.captainxq.me 上线 │
│  ├─ git push       │             │                          │
│  └─ 满7天触发周扫   │             │  CC 角色：第二大脑        │
│                     │             │  ├─ 精炼认知→insight    │
│  CC 角色：公众号写作  │             │  ├─ 回答周扫问题→question│
│  ├─ 前采→撰稿→入库  │             │  └─ 选题探讨→writing    │
│  └─ writing/协同    │             │                          │
│                     │             │                           │
│  Syncthing ← 手机   │             │                           │
└───────────────────┘             └──────────────────────────┘
```

### 两种输入线

| 来源 | 入口 | 处理 | 自动化 |
|------|------|------|--------|
| 手机 SY 碎片 | inbox/ | 本地引擎管线 | watch_inbox 检测→引擎→建站→push |
| 手机 CC 精炼/回答/选题 | 直接写 core/ 或 writing/ | 服务器 sync.sh | 每分钟 git pull→push+build |

### 关键设计原则

- **GitHub 是唯一中间人**：两端不直连，不 scp，不 ssh 互连，不 mtime 比较
- **两端各自负责**：本地管碎片管线，服务器管 CC 产出，任务天然不重叠
- **git rebase 自动合并**：两端写不同目录/不同文件名，冲突概率极低

## 核心数据流

```
手机采集（Markor 原子文件 *.md）
  → Syncthing 同步
  → kb/inbox/
  → Hermes Engine: Move → Process（AI分类+提纯+URL抓取+背景注入+打标签）→ Commit
    └─ 写入 frontmatter（结构化元数据） + 关联计算（双向写入 related）
  → core/（认知资产）│ manual/（技术知识）│ raw/（原文冷备）
  → Index Guard（增量索引）
  → Weekly Scan（摘要审计 → 生成问题 → Dashboard Active/Backlog）
  → Compiler（frontmatter.related 连通子聚类 + usage_tag 分组 + 单页≤7篇上限 → core/topic/）
  → 000_Dashboard.md（用户可见）
  → git push → 服务器 build_static → read.captainxq.me 上线
```

## 知识分流规则

### 顶层分类：硬知识 / 软智慧

AI 首先判断 `knowledge_type`：

| 类型 | 定义 | 路由 |
|------|------|------|
| hard | 工具、API、命令、代码、操作步骤 | `manual/technical/` |
| soft | 商业判断、方法论、投资理念、认知升级 | 见下方拆分 |
| mixed | 同时包含硬知识和软智慧 | 细胞分裂 + 双写 |

### soft 子级拆分（按认知分）

| 条件 | 目标 |
|------|------|
| soft + cog ≥ 3 | `core/insight/` |
| soft + cog < 3 | `core/note/` |

### mixed 双写规则

| 条件 | 目标 |
|------|------|
| mixed + cog ≥ 5 → insight 端 | `core/insight/` |
| mixed + tech ≥ 5 → tech 端 | `manual/technical/` |

### 用途标签（第二分类轴）

引擎在处理笔记时，除 `knowledge_type` 外，额外输出一个用途标签 `usage_tag`，作为独立于内容类型的第二分类轴。标签值规定如下：

| 标签 | 含义 | 示例 |
|------|------|------|
| `#tool` | 工具型资源 | 接外国手机平台、AI设计网站、软件推荐 |
| `#principle` | 认知/方法论/思维模型 | 财商的修炼、长期主义、产品思维 |
| `#case` | 具体案例/复盘 | 某产品翻车复盘、项目踩坑记录 |
| `#reflection` | 自我反思/元认知 | 执行力不足源于认知不通 |
| `#reference` | 参考资料/备忘 | 配置文档、命令备忘、速查表 |
| `#opinion` | 观点/评论/时事观察 | 消费的漩涡、高考是点人生是面 |

## 关键设计原则

| 原则 | 实现 |
|------|------|
| 事务安全 | Move → Process → Commit 三段式 |
| 可恢复 | 失败文件保留 processing，死信队列上限 3 次 |
| 原文保真 | raw/ 保存原始文件 + URL 抓取冷备 |
| 防爆设计 | 摘要级周扫（只读 `_refined.md`），1000 篇规模安全 |
| AI 只提问 | 周扫输出的是问题，不是总结 |
| 增量索引 | SHA256 对比，只更新变更文件 |
| 双策略编码 | hard用技术骨架（定位/方法/避坑），soft用认知骨架（洞察/逻辑链/金句） |
| 双轴分类 | 内容类型轴（hard/soft）+ 用途轴（usage_tag），交叉支撑编译器聚类 |
| 编译独立 | compiler.py 为独立脚本，不修改引擎已有逻辑，零回归风险 |
| 编译覆盖 | 主题页按周覆盖，每个主题永远只有一页"当前判断" |
| 往前编不往回织 | 编译器只生成/覆盖主题页，不回溯修改已有 refined |

## 检索策略

知识库检索入口分两条路径：

**首选：本地/服务器 CC 文件级检索**
- 本地 CC → 写稿前检索 KB
- 服务器 CC → 探讨时检索 KB
检索优先级：`insight/_refined.md` → `manual/technical/_refined.md` → `raw/`

**次选：静态站 read.captainxq.me 浏览**
Cloudflare Pages 静态 HTML 站，支持分类浏览和全文搜索。

| 类型 | 格式 | 示例 |
|------|------|------|
| 手机采集 | `clip_YYYYMMDD_HHMMSS.md` | `clip_20260528_224512.md` |
| 提纯版 | `{原名}_refined.md` | `clip_20260528_224512_refined.md` |
| 技术分裂 | `tech_{原名}.md` | `tech_clip_test_mixed.md` |
| 认知分裂 | `insight_{原名}.md` | `insight_clip_test_mixed.md` |

| 周扫问题 | `question_YYYYMMDD_HHMMSS_{n}.md` | `question_20260528_212715_1.md` |
| 主题页 | `topic_{主题名}.md` | `topic_ai时代人才与创作.md` |

## 精炼文件格式（Frontmatter）

每篇 `_refined.md` 使用 YAML frontmatter 记录结构化元数据：

```yaml
---
usage_tag: principle
cog_score: 8
tech_score: 0
related:
  - 财商的修炼_refined.md
created: 2026-06-13
---
```

| 字段 | 说明 | 来源 |
|------|------|------|
| usage_tag | 用途标签，取值 tool/principle/case/reflection/reference/opinion | AI 分类输出 |
| cog_score | 认知分（0-10） | AI 分类输出 |
| tech_score | 技术分（0-10） | AI 分类输出 |
| related | 关联笔记文件名列表（双向写入），指向同类 `_refined.md` | 引擎入库时计算 |
| created | 入库日期，格式 YYYY-MM-DD | 引擎写入时间 |

## 关联计算规则

入库时对每篇精炼笔记扫描已有 refined 文件，按以下规则打分：

| 条件 | 分值 |
|------|------|
| 同 `usage_tag` | +3 |
| 标题字符串重叠 ≥ 3 个字符 | +1 |

总分 ≥ 3 即视为关联，双向写入 frontmatter.related。O(n²) 比较，当前规模约 30 篇，毫秒级完成。

## 已回答问题的处理流程

```
手机 CC 连服务器 → 回答写入 core/question/ 下 ## 我的思考
  → 服务器 sync.sh 每分钟 git push
  → GitHub
  → 本地 git pull（watch_inbox 启动/触发时拉）
  → build_static.py 重建 → 静态站问题详情页可见回答内容
  → 下次 Weekly Scan →
    collect_answered_questions() 扫描已回答问题
    → 格式化后追加到审计上下文
    → AI 据此调整问题方向
```

回答不生成 inbox 笔记，只留在 question 文件本身，通过上下文间接影响后续问题。

## 本地自动化（watch_inbox.ps1）

| 机制 | 说明 |
|------|------|
| 触发 | FileSystemWatcher 监听 `kb/inbox/*.md` |
| 防抖 | 60 秒，避免 SY 同步中的重复触发 |
| 启动 catch-up | 开机/启动时先 git pull 同步服务器 CC 产出 |
| inbox 事件 | 只跑引擎（hermes_engine.py）+ build_static.py |
| 周扫 | 满 7 天触发完整管线（引擎 + 周扫 + 编译器）|
| git push | 自动 add → commit → push |
| 开机自启 | 通过 Windows 任务计划程序注册（登录时启动） |
| 互斥锁 | `.sync.lock` 防止多实例冲突 |
| 日志 | `kb/logs/watch_inbox.log` |

## 服务器自动化（sync.sh）

cron 每分钟执行：

```bash
cd /var/www/hermes-kb
git stash --include-untracked
git fetch origin
git rebase origin/main
git stash pop || true
git push origin main
cd kb/scripts && python3 build_static.py
```

## 双端 CC 指令隔离

| 环境 | 指令文件位置 | 角色 | git 跟踪 |
|------|-------------|------|---------|
| 本地 | `CLAUDE.md`、`.claude/skills/` | 公众号写作助手 | ❌ `.gitignore` 忽略 |
| 服务器 | `/var/www/hermes-kb/CLAUDE.md`、`.claude/skills/` | 第二大脑（精炼/回答/选题） | ❌ 本地 git 忽略，服务器本地保留 |

指令不通过 git 同步，各自独立更新。
