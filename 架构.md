# Hermes 架构

## 技术栈

| 层 | 选型 | 说明 |
|----|------|------|
| 语言 | Python 3.13 | 本地运行，无外部依赖框架 |
| AI | 阿里百炼 DashScope (qwen3.6-flash) | 分类、提纯、问题生成、问答 |
| 同步 | Syncthing v2.2.0 | 手机 ↔ 电脑 原子文件同步 |
| 索引 | JSON 文件 + SHA256 | 轻量增量索引，无数据库 |
| 语义检索 | sentence-transformers (bge-small-zh) | 本地嵌入模型，零成本、无 API 依赖 |

> ⚠️ 动态 Web 服务（Flask + Cloudflare Tunnel → zs.captainxq.me）已于 2026-06-15 停用。本地 API 代码保留不动，线上服务已停止+禁用自启。前端浏览全部走 read.captainxq.me（Cloudflare Pages 静态站），问答/交互走手机 CC 直连服务器。如需恢复，以管理员身份执行：

## 核心数据流

```
手机采集（Markor 原子文件 *.md）
  → Syncthing 同步
  → kb/inbox/
  → Hermes Engine: Move → Process（AI分类+提纯+URL抓取+背景注入+打标签）→ Commit
    └─ 写入 frontmatter（结构化元数据） + 关联计算（双向写入 related）
  → core/（认知资产）│ manual/（技术知识）│ raw/（原文冷备）
  → Index Guard（增量索引）
  → Weekly Scan（摘要审计 → 生成问题 → Dashboard Active/Backlog）
  → Compiler（frontmatter.related 连通子聚类 + usage_tag 分组 + 单页≤7篇上限 → core/topic/）
  → 000_Dashboard.md（用户可见）
  → 用户回答 → 写入 question 文件（不进知识库管线）
```

## 知识分流规则

### 顶层分类：硬知识 / 软智慧

AI 首先判断 `knowledge_type`：

| 类型 | 定义 | 路由 |
|------|------|------|
| hard | 工具、API、命令、代码、操作步骤 | `manual/technical/` |
| soft | 商业判断、方法论、投资理念、认知升级 | 见下方拆分 |
| mixed | 同时包含硬知识和软智慧 | 细胞分裂 + 双写 |

### soft 子级拆分（按认知分）

| 条件 | 目标 |
|------|------|
| soft + cog ≥ 3 | `core/insight/` |
| soft + cog < 3 | `core/note/` |

### mixed 双写规则

| 条件 | 目标 |
|------|------|
| mixed + cog ≥ 5 → insight 端 | `core/insight/` |
| mixed + tech ≥ 5 → tech 端 | `manual/technical/` |

### 用途标签（第二分类轴）

引擎在处理笔记时，除 `knowledge_type` 外，额外输出一个用途标签 `usage_tag`，作为独立于内容类型的第二分类轴。标签值规定如下：

| 标签 | 含义 | 示例 |
|------|------|------|
| `#tool` | 工具型资源 | 接外国手机平台、AI设计网站、软件推荐 |
| `#principle` | 认知/方法论/思维模型 | 财商的修炼、长期主义、产品思维 |
| `#case` | 具体案例/复盘 | 某产品翻车复盘、项目踩坑记录 |
| `#reflection` | 自我反思/元认知 | 执行力不足源于认知不通 |
| `#reference` | 参考资料/备忘 | 配置文档、命令备忘、速查表 |
| `#opinion` | 观点/评论/时事观察 | 消费的漩涡、高考是点人生是面 |

标签用途：
- **编译器聚类线索**：相同标签的笔记即使语义不相似也可被编译到同一主题
- **wikilink 质量**：AI 在生成 `[[wikilinks]]` 和 `[相关：xxx](xxx.md)` 时参考标签做更准确的关联判断
- **检索过滤**：支持按标签筛选知识库

## 关键设计原则

| 原则 | 实现 |
|------|------|
| 事务安全 | Move → Process → Commit 三段式 |
| 可恢复 | 失败文件保留 processing，死信队列上限 3 次 |
| 原文保真 | raw/ 保存原始文件 + URL 抓取冷备 |
| 防爆设计 | 摘要级周扫（只读 `_refined.md`），1000 篇规模安全 |
| AI 只提问 | 周扫输出的是问题，不是总结 |
| 增量索引 | SHA256 对比，只更新变更文件 |
| 双策略编码 | hard用技术骨架（定位/方法/避坑），soft用认知骨架（洞察/逻辑链/金句） |
| 双轴分类 | 内容类型轴（hard/soft）+ 用途轴（usage_tag），交叉支撑编译器聚类 |
| 编译独立 | compiler.py 为独立脚本，不修改引擎已有逻辑，零回归风险 |
| 编译覆盖 | 主题页按周覆盖，每个主题永远只有一页"当前判断" |
| 往前编不往回织 | 编译器只生成/覆盖主题页，不回溯修改已有 refined。观点变化在问答环节自然发现 |

## 检索策略

知识库检索入口分两条路径，优先级明确：

**首选：Claude Code 文件级检索**
Claude Code 通过直接读取知识库文件（.md）来理解内容，是主要检索入口。
检索优先级顺序：
1. `core/topic/`（主题页）—— 综合结论，首读
2. `core/insight/` 或 `manual/technical/` 下的 `_refined.md`（精炼版）—— 单篇深入
3. `raw/`（原文冷备）—— 事实校验
适用场景：日常写作前采、回答问题、综合判断。

**次选：静态站 read.captainxq.me 浏览**
通过 Cloudflare Pages 部署的静态 HTML 站，支持分类浏览和 Pagefind 全文搜索。
适用场景：手机端快速浏览知识库。

> ⚠️ 原先的 Flask Web 问答引擎（/ask、/chat → zs.captainxq.me）已停用。手机端问答/交互由 CC 直接连服务器替代。

**不经过的路径**：主题页不进入 CC 问答范围，避免信息冗余（主题页内容已涵盖其关联的精炼笔记）。

|------|------|------|
| 手机采集 | `clip_YYYYMMDD_HHMMSS.md` | `clip_20260528_224512.md` |
| 提纯版 | `{原名}_refined.md` | `clip_20260528_224512_refined.md` |

| 认知分裂 | `insight_{原名}.md` | `insight_clip_test_mixed.md` |

## 精炼文件格式（Frontmatter）

每篇 `_refined.md` 使用 YAML frontmatter 记录结构化元数据：

```yaml
---
usage_tag: principle
timeliness: high
cog_score: 8
tech_score: 0
related:
  - 财商的修炼_refined.md
created: 2026-06-13
---
```

| 字段 | 说明 | 来源 |
|------|------|------|
| usage_tag | 用途标签，取值 tool/principle/case/reflection/reference/opinion | AI 分类输出 |
| timeliness | 时效性，取值 high/medium/low | AI 分类输出 |
| cog_score | 认知分（0-10） | AI 分类输出 |
| tech_score | 技术分（0-10） | AI 分类输出 |
| related | 关联笔记文件名列表（双向写入），指向同类 `_refined.md` | 引擎入库时计算 |
| created | 入库日期，格式 YYYY-MM-DD | 引擎写入时间 |

兼容性：编译器同时支持 frontmatter 和旧的 `> 标签:` 行尾格式，平滑过渡。

## 关联计算规则

入库时对每篇精炼笔记扫描已有 refined 文件，按以下规则打分：

| 条件 | 分值 |
|------|------|
| 同 `usage_tag` | +3 |
| 标题字符串重叠 ≥ 3 个字符 | +1 |

总分 ≥ 3 即视为关联，双向写入 frontmatter.related。关联扫描仅在入库时执行一次，O(n²) 比较，当前规模约 30 篇量级，毫秒级完成。

## 执行约束

架构师输出执行指令前必须逐条对照。违反任一条即拆任务。

| 约束 | 阈值 |
|------|------|
| 单次修改文件数 | ≤ 5 个 |
| 单次涉及模块数 | ≤ 3 个 |
| 单次预计开发时间 | ≤ 1 小时 |
| 重构+新功能混合 | 禁止，必须拆分 |
| 一次输出指令数 | 1 条 |

禁止把大任务一次性交给操盘手。操盘手反馈后再基于实际产出输出下一条。

## v2.5 编译层设计

### 动机

现有的精炼体系（`_refined.md`）每篇笔记独立处理，跨笔记的综合判断需人手动翻阅多篇在脑中合成。编译层补上这一环：将多篇相关笔记自动合为一页"当前判断"（主题页），写稿时直接读这一页。

### 数据流位置

```
... → Engine（单篇精炼+打标签）→ Index Guard → Weekly Scan
                                  ↓
                            Compiler（语义+标签聚类 → AI 合成主题页）
                                  ↓
                            core/topic/ + 浏览页"编译论点"tab → 用户
```

### 触发时机

每周一次，接在 Weekly Scan 之后。非即时触发。

### 更新策略

覆盖制。每个主题永远只有一页 `topic_{主题名}.md`。每周编译器运行时，用截至目前所有相关笔记重新合成一版，覆盖旧文件。不保留历史版本（观点变化在问答环节发现）。

### 主题聚类策略

编译器通过双线索聚合笔记：

| 线索 | 方法 | 解决场景 |
|------|------|---------|
| 语义相似 | semantic_index 检索 top-k，余弦相似度 ≥ 0.3 | 内容相关的笔记自动成组 |
| 标签相同 | 相同 `usage_tag` 的笔记成组 | 跨内容的同用途笔记（如工具合集） |

两条线索并行，任意命中即纳入候选。AI 最终判断是否真的需要合成一个主题页，以及主题页的名称。

### 编译素材

编译器只读 `_refined.md`（精炼版），不读 `raw/`。理由：精炼版已是结构化知识，噪声低。

### 编译器输入格式

```
本周新入库笔记的 refined（变化增量）
    + 
所有已有 refined（全量，用于关联判断）
    = 
AI 判断"哪些笔记属于同一主题"→ 每组合成一页主题页
```

### 主题页内容结构

```
# {主题名}

🎯 【综合判断】
- AI 综合多篇笔记后的核心结论

🗺️ 【核心脉络】
- 该主题下的逻辑演进路径

📦 【各篇核心观点对照】
- {笔记A标题} → 核心观点
- {笔记B标题} → 核心观点

📎 【关联原始笔记】
- [{笔记A}]({笔记A_refined.md})
- [{笔记B}]({笔记B_refined.md})

> 编译于 {日期} | 覆盖 {N} 篇精炼笔记
> 标签: {tag1} {tag2}
```

### 呈现方式

浏览页新增第一个 tab"编译论点"，排在硬知识/软智慧之前。
首页不增加独立入口（用户点浏览页自然看到）。

### PC Agent 对话助手（v2.3 实施中）

浏览器内聊天界面，不建桌面壳。首页保留，加"对话"入口。

| 维度 | 设计决策 |
|------|----------|
| UI | 独立聊天页 /chat，首页加入口链接 |
| 对话能力 | 多轮上下文 + 知识库检索增强 + SSE 流式输出 |
| 检索 | 语义检索 top-3 + 相似度阈值 0.3 + 结果带 mtime |
| 会话 | 内存级 session，刷新即清，不持久化 |
| 演进发现 | prompt 检测观点变化 + 按时序展示 + 人决策是否覆盖 |
| 写入 | 暂不涉及，用户确认后写 inbox 走管线 |

改动范围：qa_engine.py + web_server.py + semantic_index.py，不新增模块，不加依赖。

### 移动端体验优化（v2.4 规划）

用户核心模式：**手机存 → 手机看 → 手机聊**，电脑仅用于特殊操作。需增强移动端完整闭环能力。

| 维度 | 当前状态 | 优化方向 |
|------|----------|----------|
| 文档删除 | 需手动文件操作 | 在浏览页/详情页添加删除按钮，支持确认后删除 + 重建索引 |
| 消息管理 | 仅支持清空整个会话 | 支持删除单条消息，保留上下文继续对话 |
| 观点演进 | 仅对话中提示变化 | 在文档详情页展示观点版本历史，支持标记/删除旧版本 |
| 我的思考 | 存在但移动端显示不明显 | 优化移动端编辑区样式，增强可见性和编辑体验 |

**改动范围**：web_server.py（新增删除接口、优化移动端样式）+ qa_engine.py（会话消息管理）

## Git 远程配置

### 双栈策略（fetch / push 分离）

| 方向 | 本地 | 服务器（阿里云） |
|------|------|-----------------|
| fetch | `ghproxy.net`（HTTPS） | `git@github.com`（SSH） |
| push | `git@github.com`（SSH） | `git@github.com`（SSH） |

原因：本地和阿里云直连 GitHub HTTPS 443 端口不稳定或不通，ghproxy 只支持 fetch 不支持 push。

### 自动同步

| 环境 | 方式 | 间隔 |
|------|------|------|
| Windows 本地 | 计划任务 `HermesKBAutoPull` → `scripts/auto_pull.ps1` | 每 10 分钟 |
| 服务器（阿里云） | cron → `sync.sh`（stash→rebase→stash pop→push） | 每分钟 |

服务器 sync.sh 使用 stash 模式而非 reset --hard，避免冲掉 Claude Code 或 CC 写入的未提交内容。

### SSH Key 管理

| Key | 用途 | 位置 |
|-----|------|------|
| `hermes-kb-server` | 服务器（阿里云）→ GitHub push | `~/.ssh/id_ed25519`（服务器） |
| `hermes-kb-local` | 本地 Windows → GitHub push | `%USERPROFILE%\.ssh\id_ed25519`（本地） |

## 已回答问题不进知识库

用户对周扫问题的回答仅写入 `question` 文件本身（`## 我的思考` 段落），不再生成 inbox 笔记进入精炼管线。理由：答案质量参差不齐，避免低质量内容污染知识库。

## 知识浏览页按 tag 分类

浏览页中知识卡片按 `usage_tag`（用途标签）分组展示，分组顺序为：

```
principle → tool → reflection → opinion → case → reference
```

同 tag 内按修改时间降序排列。标签显示中文名称：

| 标签 | 中文显示 |
|------|---------|
| principle | 💡 认知/方法论 |
| tool | 🔧 工具型资源 |
| reflection | 🪞 自我反思 |
| opinion | 🗣 观点/评论 |
| case | 📋 案例/复盘 |
| reference | 📎 参考资料 |
