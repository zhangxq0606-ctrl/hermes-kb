## 搜索策略（三级递进）

用户提问时：
1. 先用 Grep 搜索 `_refined.md` 精炼版
2. 找不到再扩大到 `manual/technical/` 和未精炼的 `core/insight/` 文件
3. 仍无结果才查 `raw/` 原文
4. 本轮无结果时告知用户知识库未覆盖，建议补充

多文档匹配时：
- 同一主题的多份 refined 优先看最新写入的
- 观点不同的在回答末尾附加冲突提醒块

## 回答格式

- 简洁直接，末尾标注引用来源（文件名）
- 根据问题复杂度自适应：简单直接回答，需展开附带逻辑链
- 标注来源时附文件名，方便用户定位

## 写作协作规范

- **启动任务先查skill**：每项任务开始时，先检查 `.claude/skills/` 下有无匹配的 skill 文件，有则按 skill 定义的工作流执行，而非凭默认行为操作。
- 用户输出主题方向后，先与用户思路探讨、延伸梳理，达成共识后再落笔成文，而非直接写出成品。探讨时聚焦核心思路，不要纠结概念定义或钻牛角尖式追问。
- **输出分支判定**：用户表明写作意图（如"写公众号"）时，讨论充分后自动在合适节点分两条线询问——① **入库**：要不要存为知识库精炼版？② **写作**：要不要输出最终正文？两者可先后、可同时，由用户决定。输出后进入**用户润色 → 用户敲定发布版 → 入库最终版**的流程。

## 归档规则

- 用户说"记下来"→ 直接存档，确认细节后写入
- 对话产生新认知 → 先问"要不要存"，用户确认再执行
- 分类标准：
  - `core/insight/`：认知深度 ≥ 3，有独立观点/洞察
  - `core/note/`：认知深度 < 3，零散灵感
  - `manual/technical/`：工具操作、技术方案
- 文件命名格式：`{主题}_refined.md`
- **原文随 refined 同进**：refined 进入目标目录时，对应的原文（`{主题}.md`）必须同步复制到同一目录，确保 `xxx_refined.md` 与 `xxx.md` 成对出现
- `_refined.md` 写完后需要用户确认

## 精炼格式

```
🎯 【一句话洞察】
<一句话直击本质>

🗺️ 【核心逻辑链】
1. <逻辑点1>
2. <逻辑点2>
...

📦 【金句与底层案例夹囊】
> <引用或案例>

[相关：xxx](xxx.md)

> ⏱️ 时效性: high|medium|low
```

## 维护（三级分工）

**A. 被动触发（自动）**
- 观点冲突：对比同一主题多篇文档，出入处主动提示
- 时效过期：`⏱️ medium` 文档超 3 个月涉及工具/平台变化时提醒
- 精炼缺失：有原始无 refined 时提示

**B. 半自动（提醒 → 等确认 → 执行）**
- 合并冲突文档
- 补写精炼版
- 更新过期内容（链接、工具名等）

**C. 不主动做**
- 周度巡检（weekly_scan.py 覆盖）
- Dashboard 手动修改（管线自动刷新）
- api/ 目录操作

## 操作纪律

- 所有修改操作必须记录日志
- 不允许修改 raw/ 下的原始文件
- 重写已有 refined 需用户明确确认

## 引用标准

insight 文件间使用 `[相关：标题](标题.md)` 格式跨链。

raw/ 和 core/insight/ 同名文件为同一材料的原始与精炼版对照关系。
