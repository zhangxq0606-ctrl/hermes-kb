# 知行计划团队看板 · 完整产品设计文档

> 版本：v2.0 | 日期：2026-05-22  
> 技术架构：单 HTML 文件 + Supabase 云端存储 + Cloudflare Pages 自动部署

---

## 一、项目背景

### 1.1 团队情况
一支 6-7 人大学生团队，暑期并行开展 7 个公益实践项目。需要一个轻量、实时的任务看板，用于任务分配、进度追踪和阶段管理。

### 1.2 核心需求
- **实时协作**：团队成员可在任意时间、任意设备上编辑任务状态
- **零后端成本**：无需自建服务器，使用 Supabase BaaS 服务
- **自动部署**：代码托管至 GitHub，通过 EdgeOne Pages 自动部署
- **固定访问链接**：团队成员通过固定链接访问，无需登录

---

## 二、技术架构

### 2.1 技术栈

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| **前端** | 单 HTML 文件（CSS + JS 内联） | 零构建成本，直接部署 |
| **数据库** | Supabase（PostgreSQL） | 免费 BaaS 服务，提供实时订阅 |
| **实时同步** | Supabase Realtime（WebSocket） | 数据变更自动推送 |
| **托管平台** | Cloudflare Pages | 绑定 GitHub 自动部署，无需备案 |
| **CDN 加速** | Cloudflare 全球节点 | 全球加速，无需备案 |

### 2.2 数据流

```
用户操作 → 前端 JS → Supabase API → PostgreSQL 数据库
              ↓                        ↓
         实时更新 ← WebSocket ← Realtime 订阅
```

### 2.3 核心约束
- 无需登录，通过链接直接访问和编辑
- 所有操作实时同步到云端
- 支持多窗口、多设备同时编辑
- 单文件交付，无构建流程

## 三、数据库设计

### 3.1 表结构

#### projects 表（项目）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY | 项目 ID |
| name | TEXT NOT NULL | 项目名称 |
| deadline | DATE | 申报截止日期 |
| award | TEXT | 奖项信息 |

#### tasks 表（任务）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID PRIMARY KEY | 任务唯一标识 |
| project_id | INTEGER REFERENCES projects(id) | 所属项目 |
| parent_task_id | UUID REFERENCES tasks(id) | 父任务 ID（支持层级） |
| name | TEXT NOT NULL | 任务名称 |
| assignee | TEXT | 负责人姓名 |
| deadline | DATE | 截止时间 |
| status | TEXT CHECK IN ('todo', 'inprogress', 'completed') | 任务状态 |
| priority | TEXT CHECK IN ('high', 'medium', 'low') | 紧急度 |
| created_at | TIMESTAMPTZ | 创建时间 |

### 3.2 数据层级关系

```
项目 (Project)
├── 一级任务 (Task) - 如"项目申报"、"自媒体传播"
│   ├── 二级子任务 (Subtask Level 2) - 如"准备申报材料"
│   │   └── 三级子任务 (Subtask Level 3) - 如"撰写项目背景"
│   └── 二级子任务 (Subtask Level 2)
── 一级任务 (Task)
```

## 五、功能特性

### 5.1 任务管理

#### 创建任务
- 点击"添加新任务"按钮
- 输入任务名称
- 自动设置默认状态（待办）、紧急度（中）

#### 编辑任务
- 点击任务卡片任意位置可编辑名称
- 通过下拉框选择负责人、状态、紧急度
- 所有修改实时保存到 Supabase

#### 删除任务
- 点击任务卡片右上角的 × 按钮
- 确认后删除任务及其所有子任务

#### 任务排序
- 按紧急度自动排序：高 → 中 → 低
- 修改紧急度后自动重新排序

### 5.2 子任务管理

#### 层级结构
- 支持三级层级：一级任务 → 二级子任务 → 三级子任务
- 每级可无限添加子任务

#### 添加子任务
- 一级任务下可添加二级子任务
- 二级子任务下可添加三级子任务
- 点击对应"添加子任务"按钮即可

#### 删除子任务
- 所有子任务都支持删除
- 删除子任务不影响同级或上级任务

### 5.3 实时协作

#### 数据同步
- 所有操作立即保存到 Supabase
- 通过 Realtime 订阅自动推送变更
- 多窗口、多设备实时同步

#### 冲突处理
- 后保存的覆盖先保存的
- 避免多人同时编辑同一任务

### 5.4 时间管理

#### 截止时间规则
| 状态 | 显示 |
|------|------|
| 距今 > 7 天 | 正常灰色文字 |
| 距今 ≤ 7 天 | 黄色背景 + ⚠️ |
| 已逾期 | 红色文字 + ⚠️ |

## 七、交互细节

### 7.1 事件绑定

#### 状态切换
- 点击状态标签循环切换：待办 → 进行中 → 已完成 → 待办

#### 负责人变更
- 选择负责人后立即保存
- 触发 Realtime 推送

#### 紧急度变更
- 选择紧急度后立即保存
- 自动重新排序任务列表

#### 子任务编辑
- 输入框失去焦点时自动保存（blur 事件）
- 避免频繁保存导致页面闪烁

### 7.2 事件委托
- 动态添加的子任务通过事件委托绑定事件
- 避免重复绑定和内存泄漏

### 7.3 数据渲染优化
- 避免重复渲染相同数据
- Realtime 推送时检查数据是否存在
- 本地操作不触发重复渲染

## 九、部署流程

### 9.1 代码托管

1. **创建 GitHub 仓库**
   ```bash
   # 本地初始化
   git init
   git add index.html
   git commit -m "Initial commit"
   git branch -M main
   git remote add origin https://github.com/username/repo.git
   git push -u origin main
   ```

2. **后续更新**
   ```bash
   git add index.html
   git commit -m "更新说明"
   git push
   ```

### 9.2 Cloudflare Pages 部署

1. **访问** https://dash.cloudflare.com → Workers & Pages → Pages

2. **创建项目**
   - 点击「创建项目」→「连接到 Git」
   - 选择仓库
   - 分支：main

3. **构建设置**
   - 框架：其他/静态网站（Other / Static Site）
   - 构建命令：留空（单 HTML 文件无需构建）
   - 输出目录：留空

4. **部署**
   - 点击「保存并部署」
   - 获得默认访问链接：`https://xxx.pages.dev`

5. **自定义域名（可选）**
   - 在项目设置 → 自定义域 → 添加域
   - 输入自定义域名（如 `kanban.example.com`）
   - 自动配置 DNS 记录

### 9.3 自动更新
- 每次 push 到 main 分支
- Cloudflare Pages 自动检测并重新部署
- 无需手动操作

## 十、Supabase 配置

### 10.1 创建项目
1. 访问 https://supabase.com
2. 创建新项目
3. 记录项目 URL 和 Anon Key

### 10.2 创建数据表

#### 步骤 1：创建 projects 表
```sql
CREATE TABLE projects (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    deadline DATE,
    award TEXT
);
```

#### 步骤 2：创建 tasks 表
```sql
CREATE TABLE tasks (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id INTEGER NOT NULL REFERENCES projects(id),
    parent_task_id UUID REFERENCES tasks(id),
    name TEXT NOT NULL,
    assignee TEXT,
    deadline DATE,
    status TEXT DEFAULT 'todo' CHECK (status IN ('todo', 'inprogress', 'completed')),
    priority TEXT DEFAULT 'medium' CHECK (priority IN ('high', 'medium', 'low')),
    created_at TIMESTAMPTZ DEFAULT NOW()
);
```

#### 步骤 3：清空旧数据（如需重置）
```sql
DELETE FROM tasks;
DELETE FROM projects;
```

### 10.3 配置 Realtime
- 在 Supabase 控制台启用 Realtime
- 订阅 `tasks` 表的所有变更

### 10.4 前端配置
在 HTML 中替换为你的 Supabase 信息：
```javascript
const SUPABASE_URL = 'https://your-project.supabase.co';
const SUPABASE_ANON_KEY = 'your-anon-key';
```

## 十三、技术细节

### 13.1 核心代码结构

```javascript
// 1. Supabase 客户端初始化
const supabaseClient = createClient(SUPABASE_URL, SUPABASE_ANON_KEY);

    // 加载项目和任务

function subscribeToRealtime() {
    // 监听数据变更
    // 自动更新界面
}

// 4. 界面渲染

function renderProjectDetail() { /* ... */ }

// 5. 事件处理

    // 绑定所有交互事件
```

### 13.2 关键优化

1. **避免重复渲染**
   - Realtime 推送时检查数据是否存在
   - 本地操作后不立即渲染，等待 Realtime 同步

2. **事件委托**
   - 动态元素通过父元素委托事件
   - 避免重复绑定

3. ** blur 事件保存**
   - 输入框失去焦点时保存
   - 避免频繁触发

## 十四、常见问题

### Q1: 数据会丢失吗？
A: 不会。所有数据存储在 Supabase 云端，刷新页面不会丢失。

### Q2: 多人同时编辑会冲突吗？
A: 后保存的会覆盖先保存的。建议避免多人同时编辑同一任务。

### Q3: 需要付费吗？
A: Supabase 免费版足够使用，EdgeOne Pages 免费部署。

### Q4: 可以离线使用吗？
A: 不可以。需要网络连接才能读写 Supabase 数据。

### Q5: 如何备份数据？
A: 可在 Supabase 控制台导出 CSV，或通过 SQL 查询备份。
