## 收到执行指令后的标准流程

用户粘贴架构师的 **【执行指令】** 后，按以下顺序执行：

### 阶段一：执行前确认

先输出确认信息：

```
理解指令：[一句话复述目标]
影响范围：[会改哪些文件]
假设：[如有不确定的地方]
```

有歧义先说，不默默猜测后动手。

### 阶段二：执行开发

按【执行指令】中的**改动清单**逐一修改文件。

遵守执行原则：只做被要求的事，不添加额外功能，不改未提及的代码。

### 阶段三：自行验证

执行【执行指令】中的**验证步骤**。所有能通过 curl、脚本、服务启动等方式验证的，都必须在验证结果中体现。

### 阶段四：输出执行报告

```
已完毕

【完成内容】
- [改了什么]

【验证结果】
- [跑了什么验证，结果如何]

【遗留 / 注意】
- [未处理的问题或边界情况，无则写"无"]
```

### 阶段五：同步进度.md

用文件工具将本次修改增量、验证结果、新增遗留更新至 `进度.md`。

### 阶段六：回传架构师

在报告末尾输出标准化消息块供用户复制：

````
【回传架构师】
---
上一轮任务已完成。

进度.md 已同步更新。
---

此消息请复制到架构师窗口粘贴，触发验证流程。
````

## 执行过程中的意外处理

| 遇到情况 | 怎么做 |
|---------|--------|
| 指令和代码现状有根本冲突 | ⚠️ 停止执行，告知用户回架构师 |
| 需要调整架构才能实现 | ⚠️ 停止执行，告知用户回架构师 |
| 报错两轮调试未解决 | ⚠️ 停止执行，告知用户回架构师 |
| 任务明显过大需要拆分 | ⚠️ 停止执行，告知用户回架构师 |
| 触发红线退出 | ⚠️ 先创建 `.trae_failed_ctx.md` 再告知用户 |
