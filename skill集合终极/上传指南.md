# 上传指南

> 这是一份给 AI 代理的通用指南，用于将项目内容上传到个人网站。

---

## 概述

本指南适用于将任何项目的内容上传到个人网站（xiuquan-lab）。支持两种部署方式：

| 部署方式 | 说明 | 适用场景 |
|----------|------|----------|
| **自动部署** | GitHub Actions 自动构建并部署到 Cloudflare Pages | 项目本身有静态网站功能 |
| **同步上传** | 克隆 xiuquan-lab 仓库，转换内容后上传 | 项目内容需要展示在个人网站 |

---

## 零、执行前提（必须先做）

**收到上传指令后，立即执行以下步骤，无需询问用户：**

### 1. 读取项目文件，提取关键信息

按优先级依次读取（找到即停）：

| 优先级 | 文件 | 提取内容 |
|--------|------|----------|
| 1 | `架构.md` 或 `ARCHITECTURE.md` | 项目名称、技术栈、核心模块 |
| 2 | `进度.md` 或 `PROGRESS.md` | 当前状态、已完成、下一步 |
| 3 | `README.md` | 项目简介、背景 |

### 2. 自动生成元数据

从读取的内容中自动提取：

| 字段 | 提取规则 |
|------|----------|
| **title** | 项目名称，通常在架构文档标题或 README 第一行 |
| **slug** | 英文短横线命名，如「白马拼车」→ `carpooling`，无中文名则用英文项目名转小写 |
| **status** | 从进度文档提取：进行中/已完成/暂停，默认「进行中」 |
| **tags** | 从技术栈提取，如微信小程序、云开发等 |
| **summary** | 一句话概括项目目的 |
| **stack** | 从架构文档技术栈提取 |

### 3. 默认行为

- **默认上传项目介绍**（`src/content/projects/<slug>.md`）
- 如用户明确要求「复盘」或「博客」，则上传到 `src/content/blog/<slug>-retro.md`
- 无需确认，直接执行

## 第一步：确认项目类型

### 类型 A：项目本身有静态网站功能

如 hermes-kb，项目自带：
- 构建脚本（如 `build_static.py`）
- GitHub Actions 工作流
- Cloudflare Pages 配置

**部署方式**：直接 `git push`，自动部署到 Cloudflare Pages。

### 类型 B：项目内容需要展示在个人网站

如小程序项目、其他独立项目，需要：
- 将项目介绍转换为 Markdown
- 上传到 xiuquan-lab 仓库
- 通过 Astro 框架生成静态页面

**部署方式**：使用同步上传流程。

---

## 第二步：了解 xiuquan-lab 内容格式

### 文件位置

| 内容类型 | 目录 |
|----------|------|
| 项目介绍 | `src/content/projects/<slug>.md` |
| 博客/复盘 | `src/content/blog/<slug>.md` |

`<slug>` 用英文短横线命名，如 `carpooling.md`、`carpooling-retro.md`。不要中文文件名。

### 项目 Frontmatter

```yaml
---
title: "项目名称"
date: 2026-06-03
status: "进行中"          # 三选一：进行中 | 已完成 | 暂停
tags: ["标签1", "标签2"]    # 用于筛选和展示
summary: "一句话说明项目要解决什么问题"
description: "可选，SEO 摘要"  # 建议加
stack: ["技术1", "技术2"]   # 可选
links:                     # 可选，没有就整段删掉
  github: "https://..."
  demo: "https://..."
---
```

### 博客/复盘 Frontmatter

```yaml
---
title: "文章标题"
date: 2026-06-03
description: "可选，显示在博客列表页的摘要"  # 建议加
project: carpooling         # 可选，填项目的 slug，实现双向关联
tags: ["产品决策", "复盘"]   # 可选
---
```

### 正文规范

项目页覆盖四个区域：
1. **背景** — 为什么做，解决什么问题
2. **核心决策** — 关键选择及理由
3. **架构概览** — 一段话 + 可选简图
4. **当前状态** — 进行到哪了，下一步是什么

博客/复盘没有固定格式，围绕一个具体决策点或阶段写反思即可。

## 第三步：同步上传流程（类型 B 项目）

### 适用场景

- 项目内容需要展示在个人网站
- 项目本身没有静态网站功能
- 需要将内容转换后上传

### 一键执行流程

```
读取项目文件 → 提取元数据 → 克隆仓库 → 生成内容 → 推送 → 告知链接
```

### AI 代理操作清单

1. **克隆仓库**
   - 克隆位置建议：
     - 如果当前项目目录是独立项目，克隆到父目录或专门的 `workspace/` 目录
     - 避免在当前项目目录内克隆，防止目录结构混乱
     - 示例：当前项目在 `d:\hermes-kb`，克隆到 `d:\workspace\xiuquan-lab` 或 `d:\xiuquan-lab`
   ```bash
   # 示例：克隆到父目录
   cd d:\  # 或其他合适位置
   git clone https://github.com/zhangxq0606-ctrl/xiuquan-lab.git
   ```

2. **生成项目介绍**（根据步骤零提取的信息自动生成）
   - 文件路径：`src/content/projects/<slug>.md`
   - 正文结构：背景 → 核心决策 → 架构概览 → 当前状态

3. **提交推送**
   ```bash
   git add .
   git commit -m "新增项目：[项目名称]"
   git push
   ```

4. **告知用户**
   - 部署链接：`https://captainxq.me/projects/<slug>`
   - 生效时间：1-2 分钟

## 第四步：自动部署流程（类型 A 项目）

### 适用场景

- 项目自带静态网站功能
- 有 GitHub Actions 工作流配置
- 有 Cloudflare Pages 部署配置

### 执行步骤

```
确认项目有部署配置
    ↓
git push 到项目仓库
    ↓
GitHub Actions 自动触发
    ↓
构建静态文件
    ↓
部署到 Cloudflare Pages
    ↓
告知用户链接
```

### AI 代理操作清单

1. **确认配置**
   - 检查 `.github/workflows/` 目录
   - 确认有部署工作流文件

2. **直接推送**
   ```bash
   git add .
   git commit -m "内容更新"
   git push
   ```

3. **告知用户**
   - 项目部署链接（根据项目配置）

## 部署配置参考

### xiuquan-lab 配置

| 配置项 | 值 |
|--------|-----|

| 域名 | `captainxq.me` |

| 配置项 | 值 |
|--------|-----|

| 分支 | `main` / `master` |
| 部署平台 | Cloudflare Pages |
| 构建命令 | `python kb/scripts/build_static.py` |
| 输出目录 | `public` |
| 域名 | 子域名（如 `kb.captainxq.me`） |

## 已知问题

- Astro 5.18 的 `entry.id` 包含 `.md` 后缀，拼接路径时记得 `.replace(/\.md$/, '')`
- 同步上传时，确保 frontmatter 格式正确，否则构建会失败
- 项目 slug 避免中文，使用英文短横线命名

## 给你的工作流（通用版）

```
确认项目类型 → 类型 A（自动部署）或 类型 B（同步上传）
        ↓
类型 A：git push → 自动部署
类型 B：克隆 xiuquan-lab → 转换内容 → git push
        ↓
告知用户链接
```
