# AI 大工程开发工作流

> 适用场景：使用 AI 作为开发主力，独立推进中大型项目，代码基础有限、依赖 AI 做技术判断的开发者。

> ⚠️ **运行环境**：本项目运行于 **Windows + PowerShell 5**。不支持 `&&`、不支持 `head`/`tail` 等 Linux 命令。使用 `python script.py` 替代 curl 进行 API 测试。详见下方各角色规范的 PowerShell 兼容说明。

---

## ⚠️ 标记说明

本文件中的所有 **⚠️ 强制** 标记都必须落实，AI不得选择性执行或跳过。

违反强制标记即视为执行失误，必须立即纠正。

---

## 核心思想

用文件代替记忆，用分工代替混用。

两个主力窗口各司其职，两份文件持续更新，整个工作流靠文件驱动闭环，不靠人工同步。

---

## 窗口分工

| 窗口 | 职责 | 禁止事项 |
|------|------|----------|
| **架构师** | 读项目文件、分析现状、输出执行提示词 | 不在这里执行开发 |
| **操盘手** | 接收指令、执行开发、反馈结果 | 不在这里讨论方向或调整架构 |

**Claude** 仅在项目第 0 天启动阶段参与：探讨大方向、推演核心架构。日常迭代不经过 Claude。

每个窗口保持干净。遇到超出本窗口职责的问题，立即切换到对应窗口处理。

---

## 两份核心文件

均放在**项目根目录**，全程维护。

### 架构文件（分布式）

根目录仅保留 `架构.md`，记录全局规范、核心数据流骨架、跨模块约定。

局部模块的详细设计放在 `docs/` 目录下，如 `docs/架构_认证模块.md`。架构师按需读取相关模块文件。

架构发生变化时，**先改架构文件，再改代码**。

### 进度.md
记录项目当前状态。每次操盘手完成一个阶段后由操盘手通过文件工具自动直接写入。内容包括：

```
# 当前目标
一句话说明当前 Sprint / 当前阶段的核心目标

# 当前阶段
例如：认证系统 / Agent 编排 / 支付模块

# 当前阻塞
- 当前影响推进的问题
- 无则写"无"

# 本阶段待办（按优先级）
- [ ] xxx
- [ ] xxx

# 本轮完成（最新一次）
- [具体完成的修改]
- [验证结果]
- [遗留问题]
- 完成时间：[日期]

# 已完成（历史累积）
- xxx
- xxx

# 技术债
- xxx
- 无则写"无"

# 下一阶段
预计下一步推进方向
```
规则：
- ⚠️ **强制**：当前目标 必须只有一个
- ⚠️ **强制**：待办 必须按优先级排序
- ⚠️ **强制**：阻塞 和 技术债 必须分开
- ⚠️ **强制**：每轮操盘手执行完成后必须同步更新
- ⚠️ **强制**："本轮完成"记录最新一次的增量内容，"已完成"记录历史累积

## 闭环节奏

日常迭代：

```
架构师 读文件 → 输出执行提示词
        ↓
  （用户复制指令到操盘手窗口）
        ↓
操盘手 执行开发 → 更新 进度.md
        ↓
  （用户复制操盘手报告的【回传架构师】到架构师窗口）
        ↓
架构师 收到回传 → 执行验证三步法 → 输出下一条（修复或推进）
  ↑                                     ↓
  └──────── 验证不通过 → 先出修复提示词 ──┘
```

⚠️ **强制**：架构师收到回传后必须执行验证三步法，不可跳过直接输出下一条。

### 架构师验证动作（强制）

每次收到【回传架构师】消息后，架构师必须执行验证三步法（详见架构师 Skill「启动约定 → 收到回传消息后的处理规则」），缺一则不下发下一条指令：读进度.md 验证结果 → 抽查关键代码 → 判断遗留。

项目启动阶段（仅有 Claude 参与时）：

```
Claude 探讨大方向 → 架构师 输出执行提示词 → 操盘手 执行 → 更新进度.md → 【回传架构师】→ 进入日常迭代闭环
```

### 每一步的具体操作

**① 架构师读文件 → 输出执行提示词**

架构师窗口**长期复用**，不关不重启。首次启动时粘贴架构师 Skill，架构师读完 `架构.md` 和 `进度.md` 后，基于你的指令输出执行提示词。后续轮次直接粘贴【回传架构师】消息即可触发下一轮，无需重复粘贴 Skill。

> ⚠️ **强制**：架构师一次只输出一条执行指令。操盘手执行完毕反馈后，架构师再基于实际产出输出下一条。禁止批量下发。

**② 操盘手执行开发**

打开新的操盘手对话，粘贴操盘手 Skill，再粘贴架构师输出的执行提示词。操盘手只执行，不决策。

遇到需要判断的问题分两种处理：
- **小问题**（语法报错、依赖冲突）→ 操盘手自行解决
- **大问题**（架构冲突、方向不确定）→ 停下来，回架构师重新评估

**窗口复用规则**：操盘手在同一窗口串行执行多个互不冲突的小任务，跨模块或换方向时新开窗口。

**③ 更新进度.md**

> ⚠️ **强制**：操盘手完成所有任务后，使用文件修改工具直接将本次完成内容、遗留问题和技术债更新至根目录的 `进度.md`，无需人工中转。

**④ 回传架构师**

> ⚠️ **强制**：操盘手更新进度.md 后，必须在报告末尾输出标准化【回传架构师】消息块。用户将此消息复制粘贴到架构师窗口，触发架构师的验证三步法。

**⑤ 架构师验证**

架构师收到【回传架构师】消息后，自动执行验证三步法（详见架构师 Skill「启动约定 → 收到回传消息后的处理规则」）。

## 任务粒度限制

⚠️ **强制**：为保证 AI 开发稳定性，单次执行任务必须满足：

- ⚠️ **强制**：最多修改 5 个文件
- ⚠️ **强制**：最多影响 3 个模块
- ⚠️ **强制**：单次开发时间控制在 1 小时内
- ⚠️ **强制**：超过范围必须重新拆分任务
- ⚠️ **强制**：大型重构必须先产出设计草稿，再进入开发

目的：
降低上下文污染、减少误改、避免架构漂移。

---

## 文档优先原则

⚠️ **强制**：任何涉及架构的变更，**先改架构文档，再改代码**。

架构师在输出执行提示词之前，若本次变更影响 `架构.md`，直接用文件工具写入，然后在输出中注明改动位置。进度.md 由操盘手执行完毕后更新，架构师不在自己窗口更新进度.md。

## 遇到大问题的处理

以下情况停止当前窗口，切换处理：

| 情况 | 在哪里处理 |
|------|------------|
| 架构需要大方向调整 | 回 Claude 探讨 |
| 指令和代码现状有根本冲突 | 回架构师重新评估 |
| 调试两轮未解决 | 回架构师分析，必要时回 Claude |
| 对下一步有多个不确定路径 | 回 Claude 讨论清楚再推进 |
| 中途遭遇重大方向调整（主动优化或被动撞墙） | 在当前或新操盘手窗口触发【上下文熔断】，重载状态；若被动撞墙，操盘手需生成失败现场快照 |

## 项目暂停与恢复

### 项目暂停时

当项目需要暂停（如切换到其他项目、等待外部依赖、时间不足等）：

1. **在进度.md中标记暂停状态**
   - 当前阻塞：项目暂停
   - 在"下一阶段"中记录暂停原因和预计恢复时间

2. **封存所有AI窗口**
   - 架构师窗口：标记为"已暂停"，不再使用
   - 操盘手窗口：标记为"已暂停"，不再使用

3. **暂停期间**
   - 不进行任何开发活动
   - 进度.md保持暂停状态标记

### 项目恢复时

当项目需要恢复开发：

1. **回到架构师窗口**
   - 架构师读取进度.md，了解暂停原因和暂停时的状态
   - 若架构师窗口已丢失，重新粘贴架构师启动.md 启动

2. **重新评估推进方向**
   - 基于暂停时的状态，重新审视架构和进度
   - 判断是否需要调整方向或继续原计划
   - 如有遗留问题，优先处理

3. **输出新一轮执行指令**
   - 架构师输出恢复后的第一条执行指令
   - 操盘手开始新一轮开发

4. **更新进度.md**
   - 清除"项目暂停"标记
   - 更新当前阻塞和待办事项

## 使用说明

### Skill 的使用方式
两份 Skill（架构师 Skill 和操盘手 Skill）**不放入项目目录**，而是每次开新对话时作为第一条消息粘贴进去。建议保存在桌面或备忘录，方便随时取用。

### 项目启动时
1. 新建项目目录
2. 在根目录创建 `架构.md`，写入项目总体设计
3. 在根目录创建 `进度.md`，写入初始阶段和目标
4. 之后按闭环节奏推进

### 窗口管理建议
- 架构师对话：**长期复用**，不关不重启。操盘手每轮执行完毕后，用户将【回传架构师】消息粘贴回架构师窗口即可继续下一轮
- 操盘手对话：同模块小任务可串行复用，跨模块或换方向时新开，执行完封存
- Claude 对话：仅在项目启动阶段使用，日常迭代不经过

## 文件结构示意

```
项目根目录/
├── 架构.md     ← 架构师必读，记录全局规范和核心骨架
├── 进度.md           ← 架构师必读，记录当前状态
├── docs/             ← 局部模块架构设计
│   └── 架构_xxx.md
├── src/              ← 源代码
└── ...
```

启动提示词保存在项目目录，每次开新窗口时粘贴：

```
知识库根目录/
├── 架构师启动.md       ← 新开架构师窗口时粘贴，AI 自动读取架构师.md
└── 操盘手启动.md       ← 新开操盘手窗口时粘贴，AI 自动读取操盘手.md
```

## 工作流的本质约束

- ⚠️ **强制**：架构师的记忆不重要，文件的更新才重要。架构师每次靠读最新文件感知项目，不依赖上一次对话。
- ⚠️ **强制**：进度.md 不更新，闭环就断掉。每次操盘手执行完必须更新，这是整个工作流唯一的强依赖。
- ⚠️ **强制**：窗口边界靠规则维持，不靠自律。两份 Skill 写明了各自的红线，遇到越界情况 AI 会提示切换。
